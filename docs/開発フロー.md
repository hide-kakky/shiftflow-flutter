# SHIFTFLOW Development Flow v1.1

最終更新: 2026-04-24

## 1. 目的
- 新要件 v1.1 に沿って、誰が実行しても同じ結果になる運用へ統一する。
- 実装前に文書の正本を揃え、実装後にテストと運用文書まで同期する。

## 2. ブランチ戦略
1. `main` に直接コミットしない。
2. `feat/*` / `fix/*` / `chore/*` / `docs/*` を使う。
3. コミットメッセージは日本語で統一する。
4. 新ブランチ作成前に `./scripts/ios_local_status.sh` を実行する。

## 3. 日常開発フロー
1. `git switch main && git pull --ff-only`
2. `git switch -c <branch>`
3. `flutter pub get && flutter gen-l10n`
4. 文書の正本を確認する
   - `docs/要件定義_v1.1_草案.md`
   - `docs/API定義.md`
   - `docs/データベーススキーマ.md`
   - `docs/Flutterアーキテクチャ.md`
5. 仕様変更時は上記文書を先に更新する
6. 実装
7. `flutter analyze && flutter test`
8. 必要な Supabase 検証
9. `plan.md / task.md / implementation_plan.md` を更新
10. コミット、PR、マージ

## 4. 優先実装順
1. DB / 権限モデル
2. API / Bootstrap 契約
3. Home / Messages / Admin の情報設計
4. 認証 / 参加導線
5. Tasks / Settings
6. テスト / 運用

## 5. セキュリティ
1. `env/*.json`, `env/*.env`, `supabase/.env` は Git 管理しない。
2. service_role / PAT をチャットやPRに貼らない。
3. 漏えいが疑われたら即ローテーション。

## 6. リリース前チェック
- `flutter analyze`
- `flutter test`
- migration 検証
- E2E 更新
- RLS / 権限 / 誤送信系回帰なし

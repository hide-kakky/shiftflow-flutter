# SHIFTFLOW Deployment Guide

## 0. 環境の考え方
- `dev` と `prod` は別DBで運用する
- データ内容は一致させない
- スキーマ / RLS / 関数 / migration 状態は最終的に一致させる
- ローンチ前の 2026-04-27 時点で、`prod` には次の migration まで適用済み
  - `20260425005000_fix_message_select_leak.sql`

## 1. リリース前チェック
```bash
flutter pub get
flutter gen-l10n
flutter analyze
flutter test
supabase db reset --local --yes
supabase db lint --local --fail-on error
```

## 2. GitHub Actions
- ワークフロー: `.github/workflows/ci.yml`
- 実行内容
  - Flutter: `analyze`, `test`
  - Supabase: migration reset + lint

## 3. Supabase反映（本番）
1. 本番プロジェクトへログイン
2. マイグレーション適用
3. Edge Functions デプロイ

例:
```bash
supabase login
supabase link --project-ref <project-ref>
supabase db push
supabase functions deploy api
supabase functions deploy dispatch_notifications
supabase functions deploy notify_due_tasks
supabase functions deploy retry_failed_notifications
```

### 3-1. FCM secrets
通知配送には、Supabase Edge Functions に次の secrets が必要。

```bash
supabase secrets set \
  FCM_PROJECT_ID=<firebase-project-id> \
  FCM_CLIENT_EMAIL=<service-account-email> \
  FCM_PRIVATE_KEY="$(cat ./secrets/firebase-private-key.pem)"
```

`FCM_PRIVATE_KEY` は改行を含む PEM をそのまま渡してよい。CI でも同じ3つを secrets として管理する。

## 5. ロールバック
- DB: ロールバック用の追補マイグレーションを作成（破壊的rollbackは避ける）
- アプリ: 直前タグへ戻して再リリース

## 6. 運用メモ
- `dev` に入れた migration は自動で `prod` に入らない
- `prod` 反映前に `./scripts/db_push.sh prod --dry-run` を実行する
- `prod` 反映には `env/db_push.prod.env` の `SUPABASE_ACCESS_TOKEN` と `SUPABASE_DB_URL` が必要
- `prod` の DB password は Supabase Dashboard の `Project Settings > Database > Settings > Database Password` で再設定できる

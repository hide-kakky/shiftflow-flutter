# SHIFTFLOW Product Discussion 2026-04-21

## 1. このメモの目的
- 2026-04-21 に行ったプロダクト議論の流れを残す。
- 実装済み機能と、今後の要件案を分けて整理する。
- 次回以降に「なぜこの方針に寄ったか」を追えるようにする。

## 2. 議論の出発点
- 原型アプリ動画を確認したところ、現行 Flutter 版よりもモバイル運用に寄った情報設計だった。
- 原型アプリでは、ホームに `直近のタスク` と `シフト` が見えていた。
- `シフト` は独立機能ではなく、メッセージの特定フォルダに投稿された情報を表示している可能性が高い。
- Flutter 版は件数サマリー中心のダッシュボードで、モバイルでは欲しい情報に届きにくいという認識で一致した。

## 3. 合意できた大枠
### 3.1 利用開始
- 自己登録する。
- 登録後に組織参加する。
- 参加後に業務機能を使う。

### 3.2 組織参加
- 組織コード検索から参加申請できるようにする。
- 招待リンク経由では、ログイン後に自動参加できるようにする。

### 3.3 構造
- `組織` の下に、親子構造を持てる `ユニット` を置く。
- `ユニット` は店舗、エリア、本部、支社、部、課、チーム、プロジェクトなどを表現する。
- `フォルダ` はメッセージや共有の運用単位として、ユニット配下に置く。

### 3.4 ナビゲーション
- モバイル下部ナビに管理を出さない。
- 管理はヘッダーメニュー内に、権限者のみ表示する。
- 中央 FAB からタスク作成とメッセージ作成を起動したい。
- 画面間スワイプ遷移は採用しない。

### 3.5 ホーム
- ホームはダッシュボード件数中心ではなく、実データ中心に寄せる。
- ユーザーごとにホーム構成を変えられるようにする。
- 初期表示はタスクとメッセージ。
- 後からフォルダやユニットを追加できるようにする。

### 3.6 タスクとメッセージの分離
- タスクは「忘れると困る」ので、現在地ユニットに依存せず全件表示がよい。
- メッセージは文脈が重要なので、現在地ユニットに依存してよい。

## 4. メッセージ画面の現在地
### 4.1 現在地ユニットの役割
- 現在地ユニットはメッセージ画面でのみ強く使う。
- ホームやタスクには無理に持ち込まない。

### 4.2 現在地ユニット切替
- 切替は 2 タップ以内が望ましい。
- モバイルではヘッダーに `現在地ユニット ▼` を表示する案を採用。
- 候補一覧は親子構造が分かる形にしたい。
- PC ではパンくず表示を採用する。

## 5. メッセージ画面のタブ構成
### 5.1 ベース案
- `全て`
- `個人`
- 現在地ユニット配下の主要フォルダ
- `親ユニット`
- `配下ユニット`

### 5.2 深い階層への対応
- 2段目、3段目タブは作らない。
- `配下ユニット` タブでは子ユニットのカード一覧を表示し、カードタップで現在地を切り替える。
- `親ユニット` タブでは親ユニット関連のカード一覧を表示する。
- PC ではパンくずで上位構造を把握しやすくする。

## 6. 一覧設計
### 6.1 メッセージ一覧
- 一覧は現在地ユニット内で閲覧可能な全件を出す。
- 状態は未読/既読などが分かるようにする。
- 色だけでなく、太字、赤丸、バッジなどで状態差を出す。
- 一覧カードには余計な情報を載せすぎない。

### 6.2 タスク一覧
- 一覧は全件表示がよい。
- どのユニットに属するタスクか分からないと文脈が失われるため、ユニット情報は見せる必要がある。

## 7. 今の論点
- `親ユニット` / `配下ユニット` の表示名をどうするか
- ユニットとフォルダを UI でどう見分けるか
- 個人メッセージを初期スコープに含めるか
- フォルダ権限をユニット所属だけで制御するか、個別許可も必要か
- ユニットロールの継承範囲をどうするか

## 8. 次回のおすすめ再開地点
1. `親ユニット` / `配下ユニット` のユーザー向け呼称を決める
2. ユニットとフォルダの見た目ルールを決める
3. 個人メッセージを初期要件に含めるか判断する
4. メッセージ一覧カードとユニットカードの最小表示項目を固める

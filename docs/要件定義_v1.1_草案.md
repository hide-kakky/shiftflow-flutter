- 作成日: 2026-04-21
- 最終更新日: 2026-04-24
- 対象プロジェクト: `shiftflow_flutter`
- ステータス: 草案
- 目的: `要件定義_v1.0` の機能要件を維持しつつ、原型アプリ動画・2026-04-21以降の議論・現行UI課題を反映して、実運用に耐える新しい要件へ統合する

## 1. この版の位置づけ
- 本書は [要件定義_v1.0.md](./要件定義_v1.0.md) の改稿草案である。
- v1.0 は「PWA 互換の移植要件」を中心に整理していた。
- v1.1 draft は「実際の利用導線」「モバイル優先の情報設計」「組織/ユニット/フォルダ構造」「安心安全な運用」を中心に再整理する。
- v1.0 の要件を捨てるのではなく、引き継ぐ要件、置き換える要件、新規追加要件を明示する。

## 2. 統合方針
### 2.1 v1.0 から引き継ぐ要件
- 認証、タスク、メッセージ、フォルダ、設定、管理、添付、通知の主要機能は引き継ぐ。
- Supabase を中心とした認可、RLS、Storage、Edge Functions の前提は維持する。
- 日本語/英語対応、CI、テスト、トレーサビリティの考え方は維持する。

### 2.2 v1.1 で置き換える要件
- `guest` 中心の仮状態管理から、`自己登録 -> 組織参加 -> 承認/参加後に利用開始` の導線へ置き換える。
- PWA 互換の画面構成から、モバイル優先の情報設計へ置き換える。
- 組織直下の単純なフォルダ運用から、`組織 -> ユニット -> フォルダ` の文脈構造へ置き換える。
- メッセージの単純な一覧中心運用から、`現在地ユニット` に基づく閲覧導線へ置き換える。

### 2.3 v1.1 で新規追加する要件
- デスクトップ版とスマホ版で明確に異なるホーム構成
- 個人メッセージの初期リリース対応
- 現在地ユニット、上位ユニット、下位ユニットを前提にしたメッセージ導線
- 誤送信、誤削除、権限事故、外部攻撃、復旧不能事故まで含めた安全要件

## 3. スコープ
### 3.1 対象
- Flutter アプリ（iOS/Android/Web）
- Supabase（Auth, Postgres, Storage, Edge Functions）
- ドキュメント、CI、テスト、運用手順

### 3.2 非対象
- Cloudflare D1/KV/R2/GAS 依存の継続
- 既存データの完全移行設計
- Google OAuth 独自実装

## 4. プロダクトの基本方針
### 4.1 目標
- ShiftFlow は、業務連絡、タスク、組織構造をひとつのアプリで扱う。
- 一般ユーザーはモバイルで素早く確認・投稿できることを重視する。
- 管理者は PC で組織・権限・構造を管理できることを重視する。

### 4.2 現時点の設計方針
- アカウントは自己登録とする。
- 登録後に組織参加を行い、参加後に業務機能を利用できるようにする。
- モバイルホームは件数中心のダッシュボードではなく、実データへ素早く届く構成を優先する。
- PC では一覧性と管理性を重視し、パンくずや広いナビゲーションを活用する。
- Notion デザインで採用した余白、情報のまとまり、整理された導線は維持する。
- ただしホームは Notion 的な情報整理を引き継ぎつつ、モバイルでは情報量を絞り、PC では一覧性を高める。

## 5. 用語
| 用語 | 定義 |
| --- | --- |
| 組織 | 利用全体を束ねる単位。会社や導入主体を表す。 |
| ユニット | 親子構造を持てる柔軟な所属単位。店舗、エリア、本部、支社、部、課、チーム、プロジェクトを表現できる。 |
| フォルダ | メッセージや共有を行う運用単位。特定ユニット配下に作成される。 |
| 現在地ユニット | メッセージ画面で現在参照しているユニット文脈。 |
| 個人メッセージ | 個人宛メッセージ。共有メッセージとは別系統で扱う。 |

## 6. 想定ユーザーと利用デバイス
### 6.1 ユーザー区分
- 一般メンバー
- ユニット責任者
- 組織管理者
- 組織参加前ユーザー

### 6.2 デバイス方針
- 一般メンバーはモバイル主軸
- 管理者は PC 主軸
- どちらのデバイスでも最低限の主要操作は可能にする

## 7. 認証・参加導線
### 7.1 認証
- FR-AUTH-001: ユーザーはメールアドレスとパスワードでログインできる。
- FR-AUTH-002: Supabase Auth が保持するセッションをアプリ再起動後に復元できる。
- FR-AUTH-003: ログイン後、所属組織とロールに基づくアクセス制御を行う。
- FR-AUTH-004: サインアウト、セッション失効、未認証状態では保護ルートから認証画面へ遷移する。
- FR-AUTH-005: セッション保持は認証情報を対象とし、表示言語などのローカル設定とは独立して扱う。

### 7.2 登録と参加
- FR-ONB-001: ユーザーは自己登録できる。
- FR-ONB-002: 登録完了後、組織参加前は限定状態として扱う。
- FR-ONB-003: ユーザーは組織コード検索から参加申請できる。
- FR-ONB-004: 管理者は参加申請を承認または拒否できる。
- FR-ONB-005: 招待リンク経由の参加をサポートする。
- FR-ONB-006: 招待リンクを踏んだユーザーは、ログイン後に対象組織へ自動参加できる。
- FR-ONB-007: `pending/suspended/revoked` など利用不可状態のユーザーは業務 API を利用できない。
- FR-ONB-008: 組織参加後に、ホーム、タスク、メッセージ、設定を利用できる。
- FR-ONB-009: 組織未参加または承認待ち状態では、業務画面ではなく参加待ち画面へ誘導する。

## 8. 組織構造と権限
### 8.1 ユニット構造
- FR-UNIT-001: ユニットは親子構造を持てる。
- FR-UNIT-002: ユニットは後から上位ユニットを追加できる構造とする。
- FR-UNIT-003: 最初は店舗単位から始めても、後からエリア、本部、全社へ拡張できる。
- FR-UNIT-004: 任意ユニットの直下にフォルダを作成できる。

### 8.2 権限
- FR-ROLE-001: 組織ロールは `owner`, `admin`, `member` を基本とする。
- FR-ROLE-002: ユニットロールは `manager`, `member` を基本とする。
- FR-ROLE-003: フォルダ単位で、閲覧、投稿、管理の権限を制御できる構造を持つ。
- FR-ROLE-004: 組織直下ユニットは `owner/admin` が作成できる。
- FR-ROLE-005: 子ユニットは `owner/admin` または親ユニット `manager` が作成できる。
- FR-ROLE-006: 新規ユニット作成者は、そのユニットの初期 `manager` になる。
- FR-ROLE-007: 親ユニット `manager` は子ユニットの管理権限を継承する。

### 8.3 フォルダ閲覧
- FR-FOLDER-001: ユーザーは、自身が権限を持つフォルダのメッセージのみ閲覧できる。
- FR-FOLDER-002: 初期リリースのフォルダ閲覧権限はユニット所属ベースで制御する。
- FR-FOLDER-003: フォルダは単なる分類ではなく、投稿・共有・閲覧の運用単位として扱う。
- FR-FOLDER-004: 個別ユーザー許可による例外的な閲覧権限は将来拡張として扱う。
- FR-FOLDER-005: フォルダ取得、作成、更新、アーカイブを行える。

## 9. ナビゲーション方針
### 9.1 全体
- FR-NAV-001: 下部ナビゲーションはモバイルの主要導線とする。
- FR-NAV-002: 下部ナビには未完了タスク数と未読メッセージ数のバッジを表示できる。
- FR-NAV-003: 下部ナビの未読メッセージバッジは、現在地ユニットに依存しない全体未読数を表示する。
- FR-NAV-004: 管理メニューは下部ナビには置かず、ヘッダーメニュー内に権限者のみ表示する。
- FR-NAV-005: 画面間のスワイプ遷移は採用しない。
- FR-NAV-006: 将来的にタブ内スワイプ切替を追加できる構造を目指す。

### 9.2 FAB
- FR-NAV-007: どの主要画面にいても、中央 FAB から新規作成を開始できる。
- FR-NAV-008: FAB から少なくともタスク作成とメッセージ作成を起動できる。

## 10. ホーム
### 10.1 基本方針
- FR-HOME-001: ホームはモバイルで「すぐ見る」ための画面とする。
- FR-HOME-002: 初期ホームはタスクとメッセージを表示する。
- FR-HOME-003: ユーザーごとにホーム構成を変更できる。
- FR-HOME-004: ホームの見た目は Notion デザインの余白、カード分割、整理された導線を継承する。
- FR-HOME-005: ただしホームは端末に応じて情報量を最適化し、同一レイアウトをそのまま拡大縮小して使わない。

### 10.2 追加できる要素
- FR-HOME-006: ホームにはタスクウィジェットを追加できる。
- FR-HOME-007: ホームにはフォルダウィジェットを追加できる。
- FR-HOME-008: ホームにはユニットウィジェットを追加できる。
- FR-HOME-009: ユニットウィジェットでは、そのユニット内で権限のあるメッセージを表示できる。
- FR-HOME-010: 複数フォルダを追加できる。
- FR-HOME-011: ウィジェットの並び順を変更できる。

### 10.3 デバイス別方針
- FR-HOME-012: モバイル版ホームは、直近のタスク、未読メッセージ、重要フォルダなど、すぐ確認すべき実データを少数に絞って表示する。
- FR-HOME-013: モバイル版ホームでは、件数サマリーや補助情報を並べすぎず、1画面内で次に取る行動が分かる構成を優先する。
- FR-HOME-014: PC版ホームは、Notion デザインの整理感を保ちながら、複数カラムや広い余白を用いて一覧性と比較しやすさを高める。
- FR-HOME-015: PC版ホームでは、モバイル版より多くの情報ブロックを同時表示できる。
- FR-HOME-016: PC版ホームでは、管理や俯瞰に向いた補助情報を表示できる。

### 10.4 初期表示ルール
- FR-HOME-017: 初期のタスク表示は期限が近い順とする。
- FR-HOME-018: 初期のメッセージ表示は未読のみ、新着順とする。
- FR-HOME-019: 追加したフォルダ表示は新着順とする。

## 11. タスク
### 11.1 一覧
- FR-TASK-001: タスク一覧は現在地ユニットに依存せず、閲覧可能な全件を表示する。
- FR-TASK-002: タスク一覧では、タスクがどのユニット文脈のものかを識別できる。
- FR-TASK-003: タスク状態は未完了、完了、期限超過などが識別できる。

### 11.2 操作
- FR-TASK-004: タスク CRUD を行える。
- FR-TASK-005: 担当者、期限、優先度、添付を設定できる。
- FR-TASK-006: タスク一覧カードには余計な情報を載せすぎず、ユニット文脈が分かる最小情報に絞る。
- FR-TASK-007: 一覧取得、自分のタスク取得、依頼したタスク取得、全件取得、詳細取得を行える。
- FR-TASK-008: 担当者割当と通知連携を行える。

## 12. メッセージ
### 12.1 基本方針
- FR-MSG-001: メッセージ画面は現在地ユニット文脈で利用する。
- FR-MSG-002: メッセージ一覧には、現在地ユニットで閲覧可能なメッセージ全件を表示する。
- FR-MSG-003: ホームでは複数ユニット・複数フォルダを横断して情報を見られる。

### 12.2 ヘッダーと現在地ユニット
- FR-MSG-004: メッセージ画面ヘッダーに現在地ユニットを表示する。
- FR-MSG-005: 現在地ユニットは 2 タップ以内で切り替えられることを目標とする。
- FR-MSG-006: モバイルでは現在地ユニット候補を階層付きリストで表示する。
- FR-MSG-007: PCではパンくずを表示し、ユニット階層を把握しやすくする。
- FR-MSG-008: メッセージ画面内の未読件数表示は、現在地ユニット文脈に応じた件数を表示できる。

### 12.3 メッセージ画面タブ
- FR-MSG-009: 上部タブには `全て`, `個人`, 現在地ユニット直下の主要フォルダ, `上位ユニット`, `下位ユニット` を配置できる構造とする。
- FR-MSG-010: `下位ユニット` タブでは、現在地ユニット直下の子ユニット一覧をカード表示する。
- FR-MSG-011: 子ユニットカードを押下すると、そのユニットへ現在地を切り替える。
- FR-MSG-012: `上位ユニット` タブでは、親ユニット関連のカード一覧を表示する。
- FR-MSG-013: `上位ユニット` タブでは、親ユニットそのものに加え、同階層の兄弟ユニットも候補として表示する案を有力とする。
- FR-MSG-014: 多段タブ化は行わず、深い階層でもタブは1段に保つ。

### 12.4 一覧表示と操作
- FR-MSG-015: 一覧は新着順を基本とする。
- FR-MSG-016: 一覧カードは最小限の情報に絞る。
- FR-MSG-017: 一覧カードには、送信者のサムネイル、件名、本文プレビュー、送信者名、日時と曜日、ピン状態を表示する。
- FR-MSG-018: 状態表現は色依存にしない。
- FR-MSG-019: 未読は太字、バッジ、アイコンなど複数手段で識別できる。
- FR-MSG-020: 既読は通常表示とし、未読との差が視覚的に分かるようにする。
- FR-MSG-021: メッセージ CRUD、コメント投稿、既読制御、ピン留め、既読状況確認を行える。
- FR-MSG-022: メッセージ作成時にフォルダ選択、テンプレート適用、添付追加を行える。
- FR-MSG-023: 一覧でフォルダ絞り込み、未読のみ表示、キーワード検索を行える。

### 12.5 個人メッセージ
- FR-MSG-024: 個人メッセージは共有メッセージと別系統で扱う。
- FR-MSG-025: 個人メッセージは初期リリースに含める。

## 13. テンプレート・設定・管理・添付・通知
### 13.1 テンプレート
- FR-TPL-001: テンプレート取得、作成、更新、削除を行える。
- FR-TPL-002: テンプレートはフォルダ文脈に応じて利用できる。

### 13.2 設定
- FR-SET-001: ユーザー設定取得・保存を行える。
- FR-SET-002: 設定対象は表示名、言語、プロフィール画像を基本とする。
- FR-SET-003: 表示言語は `ja/en` をサポートする。
- FR-SET-004: プロフィール画像の表示・更新を行える。

### 13.3 管理
- FR-ADM-001: ユーザー管理を行える。
- FR-ADM-002: 組織管理を行える。
- FR-ADM-003: 監査ログ閲覧を行える。
- FR-ADM-004: 管理画面では、組織文脈、検索、フィルター、集計、一覧、詳細を役割別に整理して表示する。
- FR-ADM-005: モバイルの管理画面では、検索、集計、一覧、詳細を段階表示し、1画面に全てを積み上げない。
- FR-ADM-006: PC の管理画面では、検索/フィルター、集計、一覧、詳細を分割表示し、俯瞰と操作を両立する。

### 13.4 添付
- FR-ATT-001: 添付メタデータ管理を行える。
- FR-ATT-002: 添付取得では署名URLなど安全な取得方法を返す。
- FR-ATT-003: `profiles`, `attachments` バケットを使用する。

### 13.5 通知
- FR-NOTI-001: 新規メッセージ通知を行える。
- FR-NOTI-002: 担当タスク作成通知を行える。
- FR-NOTI-003: 期限前日通知を行える。
- FR-NOTI-004: 重複送信を防止する。

## 14. UI/アクセシビリティ
- NFR-UX-001: モバイルでは余計な情報をカードに載せすぎない。
- NFR-UX-002: 色覚差に配慮し、色だけで状態を伝えない。
- NFR-UX-003: PCとモバイルで表示方法を変えても、概念は揃える。
- NFR-UX-004: ユニットとフォルダは UI 上で役割差が分かる見た目にする。
- NFR-UX-005: ユニットは階層を持つ所属先、フォルダは投稿先であることが一目で分かるラベル、アイコン、情報設計を採用する。
- NFR-UX-006: ユニットは組織図を連想できる表現を基本とし、フォルダはフォルダアイコンを用いて区別する。
- NFR-UX-007: 現行アプリでユーザーが慣れている、ロゴを含む共通ヘッダー、アカウントメニュー、主要画面の大見出し、カード主体の情報整理は新アプリでも継承する。
- NFR-UX-007A: 共通ヘッダー、アカウントメニュー、画面アイコン + タイトル + 説明、カードで情報を区切る見せ方は、新アプリでも基本要素として継承する。
- NFR-UX-007B: 現在の組織を画面上部で切り替える考え方は継承し、新アプリでは組織や現在地ユニットなどの文脈切替を上部に統一する。
- NFR-UX-007C: モバイルでは下部ナビ + 中央 FAB の主要導線を継承する。
- NFR-UX-008: ただし現行アプリの管理画面で見られる「縦に長い1カラムへ情報を積み上げる構成」はそのまま踏襲せず、新アプリでは役割と優先度で情報を分割する。
- NFR-UX-008A: 現行の管理画面にある「検索・集計・一覧を縦長の1カラムへ全て積む構成」は、新しい `組織 / ユニット / フォルダ / 個人メッセージ` 構造では文脈と情報量が衝突しやすいため採用しない。
- NFR-UX-008B: モバイルでは段階表示、PC では分割表示を基本とし、情報の意味ごとに表示領域を分ける。
- NFR-UX-009: モバイルでは下部ナビと中央 FAB による主要導線を維持しつつ、1画面内に置く主要アクションは原則3件以内に抑える。
- NFR-UX-010: PCでは下部ナビ前提にせず、広い画面を使って一覧、詳細、補助情報を同時に把握しやすい構成を採用する。
- NFR-UX-011: 現行アプリで有効だった「画面アイコン + 画面名 + 短い説明」の導入部は継承し、新アプリではその直下にその画面で今やるべき操作を迷わず選べる導線を置く。
- NFR-UX-012: 現在の組織や現在地ユニットなど、表示内容の文脈を決める切替要素は、画面上部の分かりやすい位置に固定して表示する。
- NFR-UX-013: フィルター、検索、再読込、集計カードは、モバイルでは折りたたみや段階表示を使って一度に見せすぎない。
- NFR-UX-014: ホーム、タスク、メッセージでは、カード型の視認性は活かしつつ、カード内情報は役割ごとに標準化し、画面ごとにバラつかせない。

## 15. 安心・安全・快適性
### 15.1 安心
- NFR-OPS-001: ユーザーは、認証された本人の権限範囲内でのみデータにアクセスできる。
- NFR-OPS-002: 他組織や権限外データが誤表示されないことを前提に、安心して利用できるアクセス制御を維持する。
- NFR-OPS-003: セッション切れ、権限不足、承認待ち、未参加などの状態は、空画面にせず状態別の案内画面で明示する。
- NFR-OPS-004: プロフィール更新、タスク保存、メッセージ投稿、既読化などの主要操作は、成功・失敗・進行中をユーザーに明示する。
- NFR-OPS-005: 削除、停止、権限変更、組織変更など影響の大きい操作は確認ステップを必須とする。
- NFR-OPS-006: 誤操作からの回復のため、取り消し、再実行、再読込、詳細確認のいずれかを主要操作後に提供する。

### 15.2 安全
- NFR-OPS-007: 組織、ユニット、フォルダ、個人メッセージのデータ境界は API、DB、UI の全レイヤーで一貫して守る。
- NFR-OPS-008: 画面に表示する操作ボタンは、ユーザー権限に応じて非表示または無効化し、押してから失敗に気づく状態を減らす。
- NFR-OPS-009: 添付ファイル、プロフィール画像などのアップロードでは、許可形式、最大サイズ、保存先の扱いを事前に表示する。
- NFR-OPS-010: 個人情報や組織情報を扱う画面では、不要な情報を初期表示しない最小権限・最小表示を基本とする。
- NFR-OPS-011: 監査対象となる重要操作は、誰が、いつ、どの組織文脈で行ったかを追跡できることを前提とする。
- NFR-OPS-012: ログイン、招待受諾、組織参加、権限変更、削除、停止、再開、公開範囲変更などの重要操作は監査ログを必須とする。
- NFR-OPS-013: 総当たり攻撃や不審な大量アクセスに備え、認証、招待受諾、検索、主要更新 API にレート制限や一時的制御を設ける。
- NFR-OPS-014: 入力値はクライアント側だけでなくサーバー側でも検証し、不正なID、危険文字列、想定外の列挙値、過大サイズを受け付けない。
- NFR-OPS-015: メッセージ本文、表示名、フォルダ名、ユニット名、テンプレート本文などの表示内容は、スクリプト実行や危険な埋め込みを防ぐ。
- NFR-OPS-016: 添付ファイルや画像アップロードでは、危険な拡張子、実行可能形式、許可外 MIME type を拒否する。
- NFR-OPS-017: 組織、ユニット、フォルダ、個人メッセージの参照先や投稿先は、現在の組織文脈と権限文脈を毎回検証し、他組織や誤文脈への混線を防ぐ。
- NFR-OPS-018: 誤送信事故を防ぐため、メッセージ投稿前に投稿先ユニット、フォルダ、公開範囲、宛先を確認できる状態を保つ。
- NFR-OPS-019: 誤削除や誤停止事故を防ぐため、復旧困難な操作には対象名を明示した確認画面を出す。
- NFR-OPS-020: 権限変更や公開範囲変更では、変更前後の影響範囲をユーザーが理解できる文言を表示する。
- NFR-OPS-021: 個人情報、組織情報、監査情報、招待情報は、ログ、画面、エクスポートに必要以上に露出させない。
- NFR-OPS-022: セッション情報、秘密鍵、サービスロールキー、内部設定値はクライアントや公開ログへ露出させない。
- NFR-OPS-023: バックアップ、復旧、障害対応の運用手順を持ち、誤更新や障害発生時に復元可能な状態を維持する。
- NFR-OPS-024: 依存ライブラリ、SDK、サーバー設定の脆弱性対応を継続できる運用を前提とする。

### 15.3 快適性
- NFR-OPS-025: モバイルの主要画面は、初回表示時にユーザーが最初に必要とする情報を1画面目で把握できることを優先する。
- NFR-OPS-026: 通常時の画面遷移、タブ切替、一覧再表示では、操作開始から 0.2 秒以内に押下反応、ローディング表示、画面変化のいずれかを返すことを目標とする。
- NFR-OPS-027: 通常時の主要画面では、操作開始から 1.5 秒以内に最初の主要コンテンツを表示することを目標とする。
- NFR-OPS-028: 読み込み中はスケルトン、プレースホルダ、進行表示などを用いて、固まって見えない状態を保つ。
- NFR-OPS-029: 検索、フィルター、再読込、タブ切替は、結果が変わったことをユーザーがすぐ認識できる形で反映する。
- NFR-OPS-030: 縦長画面では、固定ヘッダー、固定アクション、セクション分割を活用し、長いスクロールでも現在位置を見失いにくくする。
- NFR-OPS-031: 一覧画面では、0件、通信失敗、権限なし、読み込み中を見た目と文言で明確に区別する。

### 15.4 端末適応
- NFR-OPS-032: モバイルでは片手操作を考慮し、主要操作、戻る、投稿開始、既読化などの高頻度操作へ親指で届きやすい導線を優先する。
- NFR-OPS-033: PCでは、一覧と詳細、フィルターと結果、組織文脈と管理操作を分離配置し、マウス操作と俯瞰確認に適した構成とする。
- NFR-OPS-034: モバイルとPCで見た目は変えてよいが、ナビゲーション概念、用語、データ意味は一致させる。
- NFR-OPS-035: 主要画面は縦横比や端末幅の違いで情報が欠けず、極端な折り返しや操作不能領域が発生しないことを前提とする。

### 15.5 品質・運用
- NFR-SEC-001: 全テーブルに RLS を適用し、組織越境アクセスを禁止する。
- NFR-SEC-002: API は JWT 前提とし、Edge Function または同等のサーバー層でロールを検証する。
- NFR-I18N-001: 日本語・英語のUI文言を ARB で管理する。
- NFR-QA-001: CI で `flutter analyze`, `flutter test`, `supabase db reset/lint` 相当の検証を実行する。
- NFR-TRACE-001: 要件、設計、実装、テスト、運用文書をリンクで追跡可能にする。

## 16. 受け入れ基準
- AC-001: iOS/Android/Web で同じ主要 CUJ を完了できる。
- AC-002: ログイン、組織参加、タスク、メッセージ、設定、管理の主要シナリオが実施済みである。
- AC-003: 主要業務ルートが API 層経由で呼び出せる。
- AC-004: RLS テストまたは同等のアクセス制御テストで越境アクセス拒否を確認済みである。
- AC-005: アプリ再起動後に有効セッションが復元され、未認証時のみ認証画面が表示される。
- AC-006: 0件、権限不足、承認待ち、通信失敗などの代表状態で、空白画面にならず適切な案内が出る。
- AC-007: モバイル版ホームとPC版ホームで明確に情報量と配置が異なるが、概念と用語は一致している。

## 17. v1.0 との差分要約
### 17.1 維持した要件
- 認証、タスク、メッセージ、設定、管理、添付、通知の主要機能
- Supabase + RLS + API 層による認可前提
- i18n、CI、トレーサビリティ、受け入れ基準の考え方

### 17.2 置き換えた要件
- `guest` を中心とした仮状態整理 -> `組織参加前/承認待ち/参加後` の導線整理
- PWA 互換の一覧中心設計 -> モバイル優先、PC管理優先の情報設計
- 単純なフォルダ中心メッセージ導線 -> 現在地ユニット文脈を持つメッセージ導線

### 17.3 新規追加した要件
- ユニット構造、上位/下位ユニット導線
- 個人メッセージの初期スコープ
- ホームの端末別構成
- 誤送信、誤削除、権限事故、外部攻撃、復旧事故まで含めた安全要件

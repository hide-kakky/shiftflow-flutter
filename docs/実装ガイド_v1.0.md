# SHIFTFLOW Implementation Guide v1.1

- 作成日: 2026-03-21
- 最終更新日: 2026-04-24
- 対象: Flutter + Supabase

## 1. 方針
- 実装の正本は [要件定義_v1.1_草案.md](./要件定義_v1.1_草案.md) とする。
- v1.0 の「PWA 互換移植」前提は履歴として残し、実装判断は v1.1 の情報設計と権限モデルを優先する。
- 新要件の中核は `組織 -> ユニット -> フォルダ` 構造、`現在地ユニット`、`個人メッセージ分離`、`組織参加導線` である。
- UI は Notion 系の整理感を引き継ぐが、モバイルと PC で役割を明確に分ける。
- DB 変更は additive first とし、旧構造は暫定互換を持ちながら段階移行する。

## 2. 開発プロセス
1. `plan.md` にフェーズと次アクションを更新する。
2. `task.md` にタスク状態を更新する。
3. `implementation_plan.md` に実装順と前提を反映する。
4. 仕様変更は `docs/要件定義_v1.1_草案.md`、`docs/API定義.md`、`docs/データベーススキーマ.md` を先に更新する。
5. 画面仕様変更は `docs/Flutterアーキテクチャ.md`、`docs/テスト計画.md`、`docs/E2Eシナリオ.md` に追記する。

## 3. Git運用
- `main` 直コミット禁止。
- ブランチ命名: `feat/*`, `fix/*`, `chore/*`, `docs/*`。
- コミットメッセージは日本語で記述する。
- ルール参照順は `AGENTS.md` -> `docs/ルール参照順.md` -> `docs/開発フロー.md`。
- 1タスク1PRを原則とする。
- PRには `背景 / 変更内容 / 検証結果 / 影響範囲` を必ず記載する。

## 4. 実装順序
### Phase 1: 正本仕様確定
- 要件定義、API定義、DBスキーマ、Flutterアーキテクチャを v1.1 前提で同期する。
- `currentOrganization`、`currentUnit`、`message_scope`、`join_requests`、`organization_invites` の責務を固定する。

### Phase 2: DB / 権限 / API 再設計
- `units`, `unit_memberships`, `organization_invites`, `join_requests` の追加設計。
- `folders.unit_id`, `tasks.unit_id`, `messages.unit_id` と DM 文脈追加。
- RLS を `organization` だけでなく `unit` 文脈まで拡張する。
- 旧 `guest` / 組織 `manager` 前提を段階廃止する。

### Phase 3: 情報設計実装
- `Home / Messages / Admin` のモバイル版・PC版構成を実装する。
- 継承するUI:
  - 共通ヘッダー
  - アカウントメニュー
  - 画面アイコン + タイトル + 説明
  - カード分割
  - 上部の組織/文脈切替
  - モバイルの下部ナビ + 中央 FAB
- 持ち込まないUI:
  - 管理画面の縦長1カラム全積み

### Phase 4: 認証・参加・文脈管理
- 自己登録
- 組織コード参加申請
- 招待リンク参加
- 承認待ち画面
- `currentOrganization` / `currentUnit` のグローバル文脈管理

### Phase 5: 主要画面
- Home
- Messages
- Admin
- Tasks
- Settings

### Phase 6: 品質・運用
- RLS/API/UI/E2E テスト拡充
- 事故対策、監査、通知、復旧運用の文書更新

## 5. 実装規約
- Flutter
  - 状態管理: Riverpod
  - ルーティング: go_router
  - i18n: `intl` + ARB
- API
  - `ApiClient(route,args,extra)` に統一アクセスする。
  - route は暫定維持するが、引数と返却 shape は v1.1 前提へ更新する。
- DB
  - `organization_id` に加え、必要箇所で `unit_id` を持つ。
  - RLS は「自組織のみ」に加えて「許可されたユニット文脈のみ」を保証する。
  - 共有メッセージと個人メッセージはポリシーを分離する。

## 6. DoD
- コード: `flutter analyze` / `flutter test` 成功
- DB: `supabase db reset --local` / `supabase db lint --local` 成功
- 文書: 要件、API、DB、設計、テスト、運用文書のリンク整合性が取れている
- CI: GitHub Actions で上記チェックが再現できる
- 安全性: 組織越境、ユニット越境、DM越境、誤送信/誤削除事故の主要シナリオが確認済み

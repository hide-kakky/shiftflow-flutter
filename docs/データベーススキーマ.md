# SHIFTFLOW Database Schema v1.1

- 現行実装ファイル:
  - `supabase/migrations/20260321093000_init_shiftflow_schema.sql`
  - `supabase/migrations/20260326123000_notification_retry_columns.sql`
  - `supabase/migrations/20260402110000_add_fcm_notification_support.sql`
- この文書は v1.1 要件に合わせた再設計方針を定義する。DB変更は additive first を前提とする。

## 1. v1.0 からの主要差分
- 旧構造は `organizations + memberships + folders + messages/tasks` 中心だった。
- v1.1 では `organizations -> units -> folders` を中核構造とする。
- 組織ロールとユニットロールを分離する。
- `guest` 前提を廃止し、`未参加 / 申請中 / active / suspended / revoked` を中心に扱う。
- メッセージは共有メッセージと個人メッセージを区別する。

## 2. 列挙型
- 組織ロール: `owner`, `admin`, `member`
- ユニットロール: `manager`, `member`
- 参加状態: `pending`, `active`, `suspended`, `revoked`
- タスク状態: `open`, `in_progress`, `on_hold`, `completed`, `canceled`
- 優先度: `low`, `medium`, `high`
- メッセージスコープ: `shared`, `direct`
- 通知イベント:
  - `new_message`
  - `new_direct_message`
  - `new_task_assigned`
  - `task_due_tomorrow`
  - `join_request_approved`
  - `invite_accepted`

## 4. テーブルごとの見直し
### `memberships`
- 旧構造:
  - `role` に `admin`, `manager`, `member`, `guest`
- 新構造:
  - 組織ロールは `owner`, `admin`, `member`
  - 組織参加状態は `pending`, `active`, `suspended`, `revoked`
- 移行方針:
  - 旧 `manager` は `unit_memberships` 側へ段階移行
  - 旧 `guest` は未参加/申請中導線へ置き換え

### `units`
- 追加列想定:
  - `id`
  - `organization_id`
  - `parent_unit_id`
  - `name`
  - `sort_order`
  - `is_active`
  - `meta_json`

### `unit_memberships`
- 追加列想定:
  - `unit_id`
  - `user_id`
  - `role`
  - `created_at`

### `organization_invites`
- 追加列想定:
  - `organization_id`
  - `token`
  - `status`
  - `expires_at`
  - `created_by_membership_id`
  - `accepted_by_user_id`

### `join_requests`
- 追加列想定:
  - `organization_id`
  - `user_id`
  - `status`
  - `requested_at`
  - `resolved_at`
  - `resolved_by_membership_id`

### `folders`
- 旧構造:
  - 組織直下
- 新構造:
  - `unit_id` を必須追加
- 移行方針:
  - 既存フォルダは暫定的に組織直下の仮ユニットへマップする設計を検討

### `tasks`
- 旧構造:
  - `organization_id`, `folder_id`
- 新構造:
  - `unit_id` 追加
  - 表示文脈の中核として `unit_id` を使う

### `messages`
- 旧構造:
  - `organization_id`, `folder_id`, `author_membership_id`
- 新構造:
  - `unit_id` 追加
  - `message_scope` 追加
  - 個人メッセージ用の送受信文脈を追加
- 移行方針:
  - 共有メッセージとDMは同一 table でもよいが、RLS と API 契約は分離する

### `attachments`
- 追加方針:
  - `organization_id` に加えて task/message 文脈を取得時に必ず検証する
  - MIME type, size, 危険拡張子の制御を API と Storage の両方で担保する

### `audit_logs`
- 追加方針:
  - `unit_id`
  - `context_type`
  - `before_json`
  - `after_json`

## 5. RLS 方針
- 原則
  - 自組織のみ参照可能
  - さらに、許可されたユニット文脈のみ参照可能
  - 共有メッセージと個人メッセージは別ポリシーで制御する
- 補助関数見直し
  - `app.current_user_id()`
  - `app.has_membership(org_id)`
  - `app.has_org_role(org_id, allowed_roles[])`
  - `app.has_unit_role(unit_id, allowed_roles[])`
  - `app.can_read_folder(folder_id)`
  - `app.can_read_message(message_id)`
  - `app.can_post_message(unit_id, folder_id, scope)`
- 継承
  - 親ユニット `manager` が子ユニット権限を継承できることを DB 層でも担保する

## 6. Storage
- バケット
  - `profiles`
  - `attachments`
- v1.1 方針
  - 認証済みだけでなく、文脈権限を通った取得のみ許可する
  - 共有メッセージ添付と個人メッセージ添付の取得権限を分離する
  - 許可形式、サイズ上限、危険拡張子制御を追加する

## 7. 移行方針
- additive first を採用する
- 旧 `folders` のみで動く API は暫定互換期間を持つ
- 旧 `guest` / 組織 `manager` 前提は段階廃止する
- 旧ホームデータ取得 API は新 `getBootstrapData` / `getHomeContent` に寄せる
- `folder_members` は初期リリースの権限中核から外し、将来拡張扱いへ寄せる
